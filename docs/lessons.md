# 开发经验教训

## 2026-03-09: MT4 EA 开发 - 运行时安全和参数校验

### 问题描述

在静态审查 MT4 Forex Strategy Executor 时发现以下运行时安全问题：

1. **tp_levels/tp_ratios 长度不一致软失败**：只打印警告并截断，若截断后总和为 1.0 会被误判为合法
2. **参数有效期未接入主执行路径**：EA 可能在参数"未生效/已过期"时继续评估信号
3. **V1 固定品种/周期无运行时约束**：策略可能在错误品种/周期上执行
4. **必需字段缺少合法性校验**：invalid_above 和 touch_tolerance 缺失时被解析为 0，不会校验失败

### 根本原因

1. **数据一致性校验不严格**：只检查总和，未强制要求数组长度一致
2. **有效期校验未集成**：IsParameterValid() 函数存在但未在关键路径调用
3. **运行时环境假设**：假设用户会在正确图表上运行，未做强制校验
4. **字段校验不完整**：只校验了部分必需字段，遗漏了关键参数

### 解决方案

1. **tp_levels/tp_ratios 长度不一致硬失败**：
   - `ParseTPLevelsAndRatios()` 改为返回 bool
   - 检测长度不一致时立即返回 false 并设置 error_message
   - `ParseParameterJSON()` 检查返回值，失败时中止解析

2. **参数有效期接入主执行路径**：
   - `EvaluateEntrySignal()` 开头调用 `IsParameterValid()`
   - 参数无效或过期时切换到 SAFE_MODE，不评估信号

3. **V1 固定品种/周期运行时校验**：
   - `OnInit()` 中添加 `Symbol() == "EURUSD"` 校验
   - `OnInit()` 中添加 `Period() == PERIOD_H4` 校验
   - 校验失败时返回 INIT_FAILED

4. **必需字段合法性校验**：
   - `ValidateParameters()` 中添加 `invalid_above > 0` 校验
   - `ValidateParameters()` 中添加 `touch_tolerance > 0` 校验

### 经验教训

1. **数据一致性要零容忍**：
   - 关联数组长度不一致必须硬失败
   - 不能依赖"截断后恰好合法"的侥幸
   - 在解析阶段就拒绝不一致数据

2. **有效期校验要前置**：
   - 在所有业务逻辑入口检查有效期
   - 过期参数不应该被使用
   - 失败时切换到安全模式

3. **运行时环境要强制**：
   - 不能假设用户会正确操作
   - 在初始化时强制校验品种和周期
   - 不匹配时立即失败，避免错误执行

4. **字段校验要全面**：
   - 所有必需字段都要校验
   - 不能依赖"默认值 0"来判断缺失
   - 建立完整的校验清单

5. **静态审查很有价值**：
   - 可以发现运行时才会暴露的问题
   - 关注"假设"和"隐式约束"
   - 检查所有错误处理路径

### 预防措施

1. **参数校验检查清单**：
   - [ ] 所有必需字段非空校验
   - [ ] 所有数值字段范围校验
   - [ ] 所有关联数组长度一致性校验
   - [ ] 所有数组总和/比例校验
   - [ ] 有效期时间范围校验

2. **运行时安全检查清单**：
   - [ ] 品种和周期强制校验
   - [ ] 参数有效期在业务入口检查
   - [ ] K 线数量边界检查
   - [ ] 数组访问越界保护

3. **错误处理标准**：
   - 数据不一致：硬失败（return false）
   - 参数过期：切换 SAFE_MODE
   - 环境不匹配：初始化失败（INIT_FAILED）
   - 字段缺失：校验失败（error_message）

### 影响评估

**修复前**：
- 长度不一致的参数可能被误判为合法
- 过期参数仍会被用于信号评估
- 可能在错误品种/周期上执行策略
- 关键字段缺失不会被检测

**修复后**：
- 长度不一致立即拒绝加载
- 过期参数自动切换 SAFE_MODE
- 品种/周期不匹配无法启动
- 所有必需字段都有校验

## 2026-03-09: MT4 EA 开发 - 函数重复定义和数组解析

### 问题描述

在实现 MT4 Forex Strategy Executor 时遇到以下关键问题：

1. **函数重复定义**：在主文件和模块文件中都定义了 `LoadParameterPack()` 函数
2. **数组字段未解析**：tp_levels、tp_ratios、pattern 等关键数组字段被硬编码为空
3. **缺少边界保护**：直接访问 K 线数组未检查 Bars 数量
4. **路径配置不当**：默认路径与实际部署环境不一致
5. **参数热更新不安全**：加载失败会清空当前有效参数
6. **数组长度不一致校验缺失**：tp_levels 和 tp_ratios 长度不同时可能被误判为合法

### 根本原因

1. **占位实现未清理**：在模块实现完成后，忘记移除主文件中的占位函数
2. **TODO 标记被忽略**：数组解析标记为 TODO 但未及时实现，导致核心功能缺失
3. **边界条件考虑不足**：未考虑回测初期或数据不足的场景
4. **默认配置脱离实际**：使用硬编码路径而非动态获取
5. **错误处理不完善**：加载失败时直接修改全局状态，没有回滚机制
6. **数据一致性校验不足**：只校验总和，未校验数组长度匹配

### 解决方案

1. **函数重复定义**：
   - 删除主文件中的占位实现
   - 确保只在模块文件中定义函数
   - 主文件通过 #include 引入模块

2. **数组字段解析**：
   - 实现 `ParseTPLevelsAndRatios()` 函数
   - 实现 `ParsePatterns()` 函数
   - 添加数组长度和总和校验

3. **边界保护**：
   - 在所有访问 K 线数组的函数开头检查 `Bars` 数量
   - 根据需要的最大索引设置最小 Bars 要求

4. **路径配置**：
   - 使用空字符串作为默认值
   - 运行时通过 `TerminalInfoString(TERMINAL_DATA_PATH)` 获取实际路径

5. **参数热更新安全**：
   - 使用临时参数包 `temp_params` 进行加载和校验
   - 只有校验通过后才更新 `g_CurrentParams`
   - 失败时保持当前参数不变

6. **数组长度一致性硬失败**：
   - 在 `ParseTPLevelsAndRatios()` 中检测长度不一致时立即返回 false
   - 设置详细的 error_message 说明不一致情况
   - 不再使用较小长度截断，直接拒绝加载

### 经验教训

1. **模块化开发要彻底**：
   - 实现模块后立即清理主文件中的占位代码
   - 使用编译器检查重复定义

2. **TODO 要及时处理**：
   - 标记 TODO 时评估影响范围
   - 核心功能的 TODO 必须在当前任务中完成
   - 非核心功能的 TODO 可以延后但要明确标注

3. **边界条件要前置**：
   - 访问数组前先检查边界
   - 考虑极端场景（数据不足、空数组等）
   - 在函数开头统一处理边界检查

4. **配置要灵活**：
   - 避免硬编码路径
   - 使用运行时环境变量
   - 提供合理的默认值

5. **错误处理要安全**：
   - 使用临时变量进行危险操作
   - 只有成功后才更新全局状态
   - 失败时保持系统稳定运行

6. **数据一致性要严格**：
   - 关联数组必须长度一致
   - 在解析阶段就检测不一致
   - 记录详细的错误信息便于调试

7. **代码审查很重要**：
   - 静态审查可以发现编译前的问题
   - 关注函数签名冲突
   - 检查 TODO 标记的完成情况

### 预防措施

1. **开发检查清单**：
   - [ ] 移除所有占位实现
   - [ ] 处理所有 TODO 标记
   - [ ] 添加边界条件检查
   - [ ] 验证配置路径
   - [ ] 使用临时变量保护全局状态
   - [ ] 校验关联数组长度一致性
   - [ ] 运行静态代码分析

2. **模块完成标准**：
   - 核心功能完整实现（无 TODO）
   - 边界条件全覆盖
   - 错误处理完善（有回滚机制）
   - 数据一致性校验完整
   - 与主文件集成无冲突

3. **测试策略**：
   - 单元测试覆盖边界条件
   - 集成测试验证模块协作
   - 异常测试验证错误处理
   - 回测验证历史数据处理

### 影响评估

**修复前**：
- 代码无法编译（函数重复定义）
- 策略无法产生有效信号（数组为空）
- 回测可能崩溃（数组越界）

**修复后**：
- 代码可以编译
- 策略可以正确识别形态
- 回测安全运行
