# 开发经验教训

## 2025-03-09: MT4 EA 开发 - 函数重复定义和数组解析

### 问题描述

在实现 MT4 Forex Strategy Executor 时遇到以下关键问题：

1. **函数重复定义**：在主文件和模块文件中都定义了 `LoadParameterPack()` 函数
2. **数组字段未解析**：tp_levels、tp_ratios、pattern 等关键数组字段被硬编码为空
3. **缺少边界保护**：直接访问 K 线数组未检查 Bars 数量
4. **路径配置不当**：默认路径与实际部署环境不一致

### 根本原因

1. **占位实现未清理**：在模块实现完成后，忘记移除主文件中的占位函数
2. **TODO 标记被忽略**：数组解析标记为 TODO 但未及时实现，导致核心功能缺失
3. **边界条件考虑不足**：未考虑回测初期或数据不足的场景
4. **默认配置脱离实际**：使用硬编码路径而非动态获取

### 解决方案

1. **函数重复定义**：
   - 删除主文件中的占位实现
   - 确保只在模块文件中定义函数
   - 主文件通过 #include 引入模块

2. **数组字段解析**：
   - 实现 `ParseTPLevelsAndRatios()` 函数
   - 实现 `ParsePatterns()` 函数
   - 添加数组长度和总和校验

3. **边界保护**：
   - 在所有访问 K 线数组的函数开头检查 `Bars` 数量
   - 根据需要的最大索引设置最小 Bars 要求

4. **路径配置**：
   - 使用空字符串作为默认值
   - 运行时通过 `TerminalInfoString(TERMINAL_DATA_PATH)` 获取实际路径

### 经验教训

1. **模块化开发要彻底**：
   - 实现模块后立即清理主文件中的占位代码
   - 使用编译器检查重复定义

2. **TODO 要及时处理**：
   - 标记 TODO 时评估影响范围
   - 核心功能的 TODO 必须在当前任务中完成
   - 非核心功能的 TODO 可以延后但要明确标注

3. **边界条件要前置**：
   - 访问数组前先检查边界
   - 考虑极端场景（数据不足、空数组等）
   - 在函数开头统一处理边界检查

4. **配置要灵活**：
   - 避免硬编码路径
   - 使用运行时环境变量
   - 提供合理的默认值

5. **代码审查很重要**：
   - 静态审查可以发现编译前的问题
   - 关注函数签名冲突
   - 检查 TODO 标记的完成情况

### 预防措施

1. **开发检查清单**：
   - [ ] 移除所有占位实现
   - [ ] 处理所有 TODO 标记
   - [ ] 添加边界条件检查
   - [ ] 验证配置路径
   - [ ] 运行静态代码分析

2. **模块完成标准**：
   - 核心功能完整实现（无 TODO）
   - 边界条件全覆盖
   - 错误处理完善
   - 与主文件集成无冲突

3. **测试策略**：
   - 单元测试覆盖边界条件
   - 集成测试验证模块协作
   - 回测验证历史数据处理

### 影响评估

**修复前**：
- 代码无法编译（函数重复定义）
- 策略无法产生有效信号（数组为空）
- 回测可能崩溃（数组越界）

**修复后**：
- 代码可以编译
- 策略可以正确识别形态
- 回测安全运行
